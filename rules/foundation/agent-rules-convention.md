# Binding rules live in `waytide/packages/` (installed) and `waytide/rules/` (the project's own)

The project's **binding** rules — naming conventions, terminology, plan-writing norms, commit conventions, and the like — are read at the start of every session and followed. They live in two directories, split by source:

- **`waytide/packages/`** — the **installed framework packages**, each in its own subdirectory (`waytide/packages/foundation/`, `waytide/packages/language/`, …). These are installed with `git subtree` and refreshed by pulling. A package's `vocabulary.md` glossary is part of its rules — the terms are binding and can't be applied unread.
- **`waytide/rules/`** — the project's **own local rules**, one per file. The project's own conventions, not installed from a package.

- **Read every rule file under both `waytide/packages/` and `waytide/rules/` at the start of a session and follow them.** They override default behavior where they conflict; explicit user instructions still win. Read `waytide/packages/foundation/` first — it defines the framework.
- **Rules are kept separate from work artifacts.** `waytide/packages/` and `waytide/rules/` hold only rules. The project's working state lives in sibling directories under `waytide/` — `log/`, `deferred/`, `observations/`, `design/`, `plans/`, `sessions/`, `loops/`, `experiments/` — worked with as their own conventions describe, not read as binding rules at session start.
- **A rule's purpose may cut across its category.** Folder names name the *surface* a rule acts on, not the purpose it serves; judge a stance-installing rule by the mindset it installs, not by surface conformance. See the rules-install-a-mindset rule.
- **Format:** frontmatter-free markdown — a `# <title>` stating the rule, the rule in prose, then short `**Why:**` and `**How to apply:**` lines. A **framework package** rule is named by a dash-separated, lower-case name; a project's **own local rule** in `waytide/rules/` takes the ISO-8601-UTC datetime prefix (see the agent-file-names rule).
- **Recording a new local rule:** when the user states a rule, or a decision sets a rule for future work, add it as a new file in `waytide/rules/`, and add a matching one-line entry to `waytide/log/`. (A rule that belongs to a framework package is edited in that package and re-published, not added here.)

**Why:** the rules are the project's durable, in-repo conventions, committed alongside the code and read fresh each session rather than recalled — so they stay authoritative and current. One rule per file keeps each convention individually readable, referenceable, and revisable. Splitting the installed packages (`waytide/packages/`) from the project's own rules (`waytide/rules/`) keeps installed content distinct from the project's own work, so refreshing a package never touches a local rule.

**How to apply:** read the rule files under `waytide/packages/` and `waytide/rules/` at session start and hold to them. When a new binding convention is set, write it in `waytide/rules/` in the format above and log it. Prefer an observation over a rule when the thinking hasn't stabilized (see the observations rule). Related: the agent-file-names rule, the rules-install-a-mindset rule, and the decision-log rule.
