# `waytide/rules/` — binding project rules, one per file

`waytide/rules/` holds the project's **binding** rules — naming conventions, terminology, plan-writing norms, commit conventions, and the like — one rule per file. They arrive two ways: installed framework packages, each in its own subdirectory (`waytide/rules/foundation/`, `waytide/rules/language/`, …), and the project's own **local** rules, as loose files at the top level of `waytide/rules/`. A package's `vocabulary.md` glossary is part of its rules — the terms are binding and can't be applied unread.

- **Read every rule file under `waytide/rules/` at the start of a session and follow them.** They override default behavior where they conflict; explicit user instructions still win. Read `waytide/rules/foundation/` first — it defines the framework.
- **Rules are kept separate from work artifacts.** `waytide/rules/` holds only rules. The project's working state lives in sibling directories under `waytide/` — `log/`, `deferred/`, `observations/`, `design/`, `plans/`, `sessions/`, `loops/`, `experiments/` — worked with as their own conventions describe, not read as binding rules at session start.
- **A rule's purpose may cut across its category.** Folder names name the *surface* a rule acts on, not the purpose it serves; judge a stance-installing rule by the mindset it installs, not by surface conformance. See the rules-install-a-mindset rule.
- **Format:** frontmatter-free markdown — a `# <title>` stating the rule, the rule in prose, then short `**Why:**` and `**How to apply:**` lines. The filename follows the ISO-8601-UTC prefix convention (see the agent-file-names rule).
- **Recording a new local rule:** when the user states a rule, or a decision sets a rule for future work, add it as a new loose file at the top level of `waytide/rules/`, and add a matching one-line entry to `waytide/log/`. (A rule that belongs to a package is edited in that package and re-published, not added here.)

**Why:** the rules are the project's durable, in-repo conventions, committed alongside the code and read fresh each session rather than recalled — so they stay authoritative and current. One rule per file keeps each convention individually readable, referenceable, and revisable. Keeping them in `waytide/rules/`, apart from the working-state directories, means the read-at-session-start set is exactly one directory.

**How to apply:** read the rule files under `waytide/rules/` at session start and hold to them. When a new binding convention is set, write it here in the format above and log it. Prefer an observation over a rule when the thinking hasn't stabilized (see the observations rule). Related: the agent-file-names rule, the rules-install-a-mindset rule, and the decision-log rule.
