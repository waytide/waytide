# Review the website content

Review the drafts in `docs/` before the documentation site is built out. So far: `capabilities.md` (the capability list), `experiments.md`, and `human-in-the-loop-design.md`, plus whatever pages are added next.

Check each page for:

- **Accuracy** — does it match what the packages actually do and say? (The rules are the source of truth.)
- **Plain language** — no cute or clever framing, no unexplained jargon or abbreviations, real names for things. (Several fixes already made mid-draft; a clean read-through is still owed.)
- **Capability-first framing** — leads with what the reader gets and how it works; leaves packages/installation/distribution for a later "how to get it" section.
- **Consistency** — the pages read as one voice and follow the same shape (what you get → how it works → worked example → where it lives).
- **Completeness** — the capability list has a page or section for each capability; nothing important is missing.

**Do this before:** building out the full site.

**Why:** the drafts were written quickly and needed several plain-language corrections during drafting; a deliberate review pass will catch the rest and confirm the pages are accurate and consistent before more are written on top of them.

**How to do it:** read each `docs/` page against the packages it describes and against the plain-language rules (`communicate-plainly-name-the-thing`, `no-slang`, `expand-abbreviations-on-first-use`); fix what's off; confirm the capability list and the pages line up.
